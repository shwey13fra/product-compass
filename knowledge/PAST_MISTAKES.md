# Past Mistakes — Product Compass

> When something bites us, record the *class* of error and the pattern that prevents it.
> Not the one-off bug — the lesson. Newest at top.

- 2026-07-15 · **A DB migration and its code deploy are ONE atomic change. Dev and prod share one Supabase project — so running a migration "locally" ships it to prod instantly, while prod keeps serving the old code.** Stage 11's migration dropped the permissive policies on `applications`; the Stage 11 code that talks to the new RPCs sat uncommitted for a day. Vercel kept serving `.from("applications")` against a now-deny-all table → `{data: [], error: null}` → live `/tracking` silently showed "Nothing tracked yet" to every user, with no error box and nothing in the logs. The migration file even said "run this BEFORE (or **together with**) deploying the app code" — we did the "before" and never closed the loop. Insidious detail: the anon-key `[]` in the RLS proof **was** prod's behaviour; it was read as a security pass, so the breakage hid inside the evidence that was supposed to find it. *Pattern:* (1) a stage isn't done until **migration + deploy are both live** and verified **against the prod URL**, not localhost; (2) when a migration is backward-**incompatible**, deploy the code first or in the same push — never migrate and walk away; (3) `git show HEAD:<file>` vs the working tree answers "what is actually running in prod?" in one command — run it before trusting any prod verification; (4) with a shared dev/prod DB there is no such thing as a "local" migration.

- 2026-07-15 · **Two subsystems can key off two different "identities" — don't assume the id you can see is the id being metered.** Seeding `ai_usage` to the cap didn't trigger the refusal. Root cause (from the evidence, not the first guess): the uid was taken from `applications.owner_key` = a **`compass_uid`**, but the tester was **signed in**, and `resolveIdentity()` prefers the **verified auth user id** over `compass_uid`. Different namespaces — seeding the compass_uid could never have fired, and `ai_usage` proved it by growing a *second* row (`count = 1`) under the auth id seconds later. The quota was correct throughout; we were reading the wrong row, and the "failed" test spent real credit. *Pattern:* for any per-identity meter, make one real call, then **read the identity back from the store** (`select identity, count from ai_usage order by updated_at desc`) and seed **that** value — never derive it from a different table's key. Same class as the RLS-silent-empty bug: a fallback chain (auth id → `compass_uid` → `ip:`) resolves to a key you didn't predict, so it presents as "the feature didn't fire" instead of "you tested the wrong row". *Separate, still-true hazard (a hypothesis in this incident, not the proven cause — never confirmed which origin `a16e565d` belonged to):* `localStorage` is **per-origin**, so `localhost:3000` and the lilac URL hold different `compass_uid`s = different trackers and different quota buckets. An empty `/tracking` on prod after testing on localhost is expected, not a regression.

- 2026-07-15 · **An empty result only proves denial if you separately prove the rows exist.** `select * from applications` with the anon key returning `[]` is exactly what a *passing* RLS deny-all and an *empty table* both look like — the same trap as the 2026-06-28 missing-SELECT-policy bug, just inverted (there, `[]` hid a bug; here, `[]` could have hidden a non-test). *Pattern:* every isolation proof needs three legs, not one — a **control** (the key works at all: read a public table), the **attack** (denied, ideally targeting a real known key so it can't be dismissed as "no matching rows"), and the **app path** (the same client reads the rows via the intended door). Only the third leg makes the second one mean anything. See `docs/VERIFICATION_STAGE11.md` §b.

- 2026-06-28 · **Tailwind v4 can't generate classes from interpolated strings** (`` `bg-${tone}-soft` ``, `` `text-${x}` ``). The JIT scans source for *complete* literal class names at build time; a runtime-built string isn't found, so the class never ships and the element renders unstyled — no error, just missing styles. *Pattern:* map a variant to a full static class string (`const TONE = { success: "bg-success-soft text-success", … }`) and interpolate the whole string, never a fragment. Caught in `FitReadView` before it shipped.

- 2026-06-28 · **RLS-enabled table with no SELECT policy → anon reads return empty with NO error (silent "no data").** User inserted 50 rows via the SQL editor (service-role, bypasses RLS) but the app (anon key) kept showing "No roles yet". `select` returned `{ data: [], error: null }` — looks identical to a truly empty table. *Pattern:* when an anon query returns 0 rows + no error but you expect data, suspect a missing RLS SELECT policy before suspecting the insert. For shared/public tables add `create policy ... for select to anon using (true)` (see `scripts/roles-rls-policy.sql`); keep writes service-role only.

- 2026-06-28 · **A "seeded" claim in the journal isn't proof — verify against the live DB.** Journal said 50 roles were seeded, but `select count(*)` on `roles` returned 0. The seed had never landed (or was run against a different project). Compounding it: RLS *is* enabled on `roles` (anon `SELECT` allowed, `INSERT` denied), so seeding via the anon key/app is impossible — `node` insert failed with "violates row-level security policy". *Pattern:* (1) verify data state with a real query before building on it; (2) seed shared/RLS-protected tables via a SQL file run in the Supabase SQL editor (service-role, bypasses RLS), never via the anon client. See `seed.sql` + `scripts/gen-seed-sql.mjs`.

- 2026-06-27 · **Pasted env values that include extra path/suffix break the client.** Supabase URL was pasted as the REST endpoint (`…supabase.co/rest/v1/`) but `supabase-js` wants the bare project URL and appends `/rest/v1/` itself → doubled path, silent failures. *Pattern:* when a user pastes a connection value, sanity-check it against what the SDK expects (bare host, no trailing slash/path/quotes/whitespace) before trusting it.
