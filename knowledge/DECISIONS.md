# Decision Log — Product Compass

> One line per major architectural choice. Newest at top. Format: date · decision · why.

- 2026-07-15 · Stage 12 cron writes to `roles` as a **dedicated Supabase auth bot user whose JWT is forwarded to the existing `runIngest(client)`** — NOT a `SECURITY DEFINER ingest_upsert_roles` (the originally specified design) · the cron has no admin JWT and service-role is banned, so such a function could only be granted to **`anon` — whose key is public** (it ships in the browser bundle) → world-callable, and `SECURITY DEFINER` bypasses the roles admin-write RLS by design. A `CRON_SECRET` check in the route is no defence: an attacker POSTs straight to `/rest/v1/rpc/…` and never touches Next.js (proven — a terminal `curl` with only the anon key called `rpc/get_applications`). The bot keeps **one** write path shared with the admin button and adds zero attack surface. If ever revisited, the secret **must** be an argument the function itself verifies, as Stage 11's uid-bearer RPCs do.
- 2026-07-15 · The ingest bot gets its own **`is_ingest_bot()`** predicate; `roles` write policies become `is_admin() or is_ingest_bot()` — the bot is **not** added to `is_admin()`/`ADMIN_EMAILS` · least privilege: `ADMIN_EMAILS` (`config.ts`) drives only the **UI** while RLS is the real gate, so an `is_admin()` bot with a leaked password would be full admin **at the data layer** (read `referral_applications`, override statuses) with the UI irrelevant. Bot can write `roles` and nothing else; `delete` stays admin-only since expiry flips `is_live` and keeps history.
- 2026-07-15 · Expiry is **per-source and skipped entirely for any source that errored**, plus a **circuit breaker** when a source returns 0 rows while it still has live ones · fixes a real Stage 8 bug — `classifyExpiry` compared all ingested ids against the current pull with no notion of source, and `fetchAllSources` is best-effort, so an Adzuna outage flipped `is_live=false` on every Adzuna role. Harmless when a human clicks Sync and reads the errors; **silent** under a daily cron. Per-source scoping alone is insufficient (a failed pull has an empty fresh set). Failure mode is now stale-and-visible rather than silent data loss.
- 2026-07-15 · Dedupe keeps **exact** `(company,title,location)` matching with a **richer-JD tiebreak** (fallback: source rank); fuzzy similarity **rejected** · a false merge silently **hides a real role**, a missed merge only shows a duplicate — asymmetric costs, and the duplicate is cosmetic while the hidden job breaks the product's core promise. The `SOURCE_ORDER` fallback is load-bearing: stable sort would otherwise let input order decide ties.
- 2026-07-15 · Cron instrumentation uses a new **`trackServer()`**, not `track()` · `track()` returns early when `typeof window === "undefined"`, so the originally specified `track('ingest_run')` from a cron would have been a **silent no-op** — instrumentation that never fires behind a dashboard you'd trust. Mirrors `logError`, which works server-side precisely because it lacks that guard.

- 2026-07-14 · Stage 11 anonymous `applications` isolation = **RLS deny-all + SECURITY DEFINER RPCs that require the uid** (`get/get/upsert/backdate_application(p_uid,…)`), NOT the `current_setting('request.headers')` header approach · the header approach depends on Supabase's gateway forwarding a custom header into Postgres, which can't be reproduced locally → a prod-only silent-break/leak risk; the RPC approach is deterministic, has no gateway dependency, and `owner_key` was already threaded as `ownerKey` everywhere. Enumeration impossible: you can only touch rows whose secret uid you supply.
- 2026-07-14 · Stage 11 AI budget = **durable Postgres quota** (`increment_ai_usage` SECURITY DEFINER, monthly per identity=auth-id-else-compass_uid, `AI_MONTHLY_LIMIT`=15) + **hourly per-IP backstop** (`check_ip_rate`, `AI_IP_HOURLY_LIMIT`=10), replacing the per-process in-memory counter · anon key only (no service-role): definer fns can't be raced/spoofed. IP backstop catches compass_uid rotation. Quota-check failure fails **CLOSED** (503 → free manual path); IP-check failure fails **open** (quota is the real guard). Validation-400s consume neither.

- 2026-07-14 · Stage 10 error monitoring = a **server-side `errors` Supabase table**, NOT Sentry · zero-cost/no-paid + Hobby-friendly: reuses the existing insert-only RLS pattern, adds ZERO client bundle, and captures the critical `/api/position` Anthropic failures server-side where we control the catch blocks. Tradeoff: no auto-capture of unhandled browser exceptions (`// TODO(v2): Sentry` if needed).
- 2026-07-14 · Stage 10 analytics = **first-party Supabase `events` table** (insert-only RLS, client can INSERT but never SELECT/UPDATE/DELETE; reads only via dashboard/service-role) + **Vercel Analytics** (free) for page traffic · zero paid tools; `track()` is fire-and-forget, never throws, never blocks UI. Hard PII rule: props carry ids/enums/booleans only — never emails, JD/experience, or brief/message text.
- 2026-07-14 · Added a minimal **"Copy brief"** button to the brief card (the only new UI in Stage 10) · `brief_copied` needed a real copy action to instrument, and copying the brief to use it is an obvious product gap.
- 2026-07-13 · `getRoleById` maps a malformed (non-uuid) id / Postgres `22P02` to the **notFound()** path · a malformed id can never match a role; previously it rendered the error card and leaked the raw DB error string into the HTML.

- 2026-07-07 · Stage 8 ingest writes to the RLS-locked `roles` table by **forwarding the signed-in admin's Supabase JWT** to `POST /api/ingest` (per-request client carrying `Authorization: Bearer`), NOT a `service_role` key · honors the existing admin-write RLS + the "anon key only, no service-role" rule; the route re-checks `isAdminEmail()` server-side before writing.
- 2026-07-17 · Fit read returns `matchPct: number | **null**` — `null` = "the JD is too thin to score", rendered as an honest can't-score card, never a number · a ratio whose denominator can collapse to 1 is a coin flip, not a score (a 131-char JD scored a real career 10% and the word "dau" 95%). `MIN_JD_THEMES = 3`. Making "can't tell" representable in the **type** (not `0`, not a sentinel) forces every caller to handle it and lets `tsc` find them.
- 2026-07-17 · Theme keywords match at a **word start**, and keywords ≤3 chars also need a closing boundary (`\bais?\b`) · `includes("ai")` fired on "failed"/"email"/"maintain"/"domain". Long keywords stay prefix matches so "interview"→"interviews" and "prioriti"→"prioritisation" keep working.
- 2026-07-17 · `brief_feedback` is **RLS deny-all + uid-scoped `SECURITY DEFINER` RPCs**, never a "uid = owner" policy · `compass_uid` is a localStorage value the client *asserts* and the anon key is public, so such a policy is satisfied by claiming someone else's uid. A policy can only enforce what the DB can verify. Same pattern as `applications` (stage 11). Admins read via a separate `is_admin()` select policy (the `sync_runs` shape).
- 2026-07-17 · `StoredBrief.mode` is **optional at version 1**, and briefs predating Stage 13 report `'unknown'` rather than a guessed `'live'` · `loadBrief` guards `version !== 1`, so bumping the version to add a field would silently wipe every saved brief on every device. A guessed mode would also corrupt the live-vs-manual comparison that is the whole point of `/admin/quality`.
- 2026-07-17 · `brief_feedback.rating` is **nullable** · the "did you use the brief?" prompt fires on apply for users who never rated, so a row may carry only `used_in_application`. A NOT NULL rating makes that row unwritable.
- 2026-07-17 · Admin-posted referral roles set `source = 'referral'` explicitly · the column has no default, so omitting it wrote NULL and Stage 8's `where source is null` backfill swept referral roles into the `'seed'` bucket — where the documented cleanup (`delete from roles where source='seed'`) would have deleted the warm path. Comment in `stage8-job-ingestion.sql` defused.

- 2026-07-16 · The cron endpoint returns **two distinguishable 401 bodies** — `"Not configured."` (CRON_SECRET unset → fail closed) vs `"Unauthorized."` (comparison failed) · a single opaque 401 makes "the gate rejects" and "the gate is misconfigured" indistinguishable from outside. In practice this **proved the Vercel env var was set** without anyone handling the secret. Costs one branch; leaks nothing (an attacker learns only whether the endpoint is configured, and it rejects them either way).
- 2026-07-16 · `LEVER_COMPANIES` stays **empty** (`config.ts:25-27`) — Lever reports `0 fetched, ok: true` every run and this is **correct, not a bug** · nothing is configured to fetch, so `previouslyLive = 0` and the circuit breaker rightly stays quiet. Add a slug to turn it on. Recorded because an unconfigured source and a failing one look identical in a run summary.

- 2026-07-07 · Ingested roles are scored by a **new pure-JS rule-based scorer** (`src/lib/realPmScore.ts`), never an AI call · protects the €5 Anthropic budget (a sync is free); rougher than the hand-authored seed scores, so the source badge keeps it honest.
- 2026-07-07 · The new `source` column double-duties as the **"dummy data" marker**: seed rows tagged `source='seed'` (badged "Sample", deletable), ingested rows `'greenhouse'|'lever'|'adzuna'` · one column distinguishes sample vs real and gates the expire pass (`.in('source', INGESTED_SOURCES)` never touches seed/referral rows).
- 2026-07-07 · Ingested roles are **cold-path**: Apply links OUT to `apply_url` (no fake apply), fit read + crowd stat only; admin referral roles keep the warm thread · legal sources give a real apply URL, and warm collaboration only makes sense for admin-tagged referrals. Adzuna capped at 50 rows/sync (cost safety). `// TODO(v2): Vercel Cron` for daily auto-sync.
- 2026-07-01 · Sign-in uses a **typed OTP code** (`verifyOtp`), not the magic link · magic links broke in practice — Gmail pre-scans/consumes the one-time link and PKCE needs the same browser; a typed code is immune to both. Link kept as a fallback. App accepts 6–10 digits (Supabase OTP length is configurable; it emits 8).
- 2026-07-01 · Supabase **"Confirm email" turned OFF** + **Gmail custom SMTP** · redundant for passwordless OTP (the code proves ownership); with it ON, new users got the code-less "Confirm signup" template so no email except the pre-existing one worked. Custom SMTP was also required to edit templates (add `{{ .Token }}`) and lifts the send limit.
- 2026-07-01 · Referral status is **role-permissioned** (referrer drives Seen/Shared/Shortlisted/Closed; referee only Closed; admin overrides) via `allowedStatusesFor()` + `StatusStrip` `allowed` prop · maps status to who actually has the information; UX guardrail only (RLS still gates row access). Anonymous `/tracking` strip stays all-settable (it's your own tracker).
- 2026-07-01 · Landing page is the **shared entry for everyone with an `AuthNav` header** (no auto-redirect for admins); landing sign-in returns to `/roles` · one predictable entry point; the Admin link in the nav is the discoverable path. Repo kept **public + personal account** so Vercel Hobby deploys free (no Teams/Pro).
- 2026-06-30 · Stage 7 auth = **two tracks**: existing anonymous `applications`/`compass_uid` + public `roles` read stay untouched; only the NEW referral tables (+ `roles` writes) get auth-based RLS · the only way to satisfy "turn ON RLS" AND "keep the user side working without login."
- 2026-06-30 · Admins identified in Postgres by a `public.is_admin()` SQL function holding the same email list as `ADMIN_EMAILS` in `src/config.ts` (two places, kept in sync) · simplest v1 enforcement; no extra tables/triggers/flags. `config.ts` drives the UI, `is_admin()` drives RLS.
- 2026-06-30 · `comments` RLS references ONLY referee/referrer, never `is_admin()`, so admins are **structurally** unable to read/write comments; a SECURITY DEFINER trigger keeps `comment_count`/`last_comment_at` on `referral_applications` so admins can still see *that* a thread exists · meets "admin sees status + thread-exists but not contents" without trusting app code.
- 2026-06-30 · Referrer is matched by **email** (`role.referrer_email` = signed-in email), not a pre-created account · referrer can be tagged before they ever sign in; no invite plumbing in v1.
- 2026-06-30 · Auth is **client-side** (magic link, `@supabase/supabase-js` `detectSessionInUrl` + PKCE); admin/referral pages are client components gated on the session · matches the existing all-client tracking pattern; avoids `@supabase/ssr` cookie plumbing. Anon key only — no service-role anywhere.
- 2026-06-28 · Live positioning reuses the SAME `buildPositioningPrompt` + `parseBrief` as the manual path (route sends `{role, profile}`, assembles + parses server-side) · one prompt + one parser = live and manual briefs are identical-shaped; less to keep in sync.
- 2026-06-28 · Budget guard = per-process module counter (default 15, `POSITION_CALL_CAP`) + `claude-haiku-4-5` + `max_tokens 1024`; validation 400s don't increment · cheap + bounded; resets on cold start (a dev guard, not a per-user quota — plumb `compass_uid` later if needed).
- 2026-06-28 · Fit read is pure-JS theme bucketing (JD keyword themes ∩ experience → % match + framable gaps), NOT an AI call · must be free/instant and available the moment experience is filled, before any brief.
- 2026-06-28 · Stage 3 briefs persist per role in localStorage (`compass_brief:<roleId>`), experience in `compass_experience` · no DB needed for v1 positioning; matches the no-auth `compass_uid` model.
- 2026-06-28 · Personalisation (Stage 2.5) is SSR-first: `prefs` starts null so server + first client render show browse mode, then localStorage loads and the list re-ranks · keeps roles server-rendered + avoids hydration mismatch; brief re-rank flash for returning users is acceptable.
- 2026-06-28 · Match score is pure JS (no AI): archetype 50 (dominant) + location 20 + real_pm_score 0–30 + small freshness boost; disguised roles (<40) capped at "partial" fit · scoring must be free/instant, and a disguised role must never be sold as a top match (product thesis).
- 2026-06-28 · Seed data = single source `scripts/roles-data.mjs` → `gen-seed-sql.mjs` emits `seed.sql`, run in Supabase SQL editor; `roles` gets a public SELECT-only RLS policy · anon can read shared roles, but writes/seeding stay service-role only.
